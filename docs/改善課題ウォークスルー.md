# 改善課題ウォークスルー

更新日: 2026-07-22

対象: shonan-dx-app の公開版（app.shonan-dx.com）

目的: これから修正する課題を、利用者・運営管理者・UI/UX・技術運用の視点で優先順位付けし、実装順を決める。

## 1. 結論

現状は、Googleログインと会社別プラン管理だけが本番Supabaseへ接続され、顧客・品目・見積・請求書・更新履歴はブラウザのlocalStorageで動くデモ段階である。

したがって、複数の契約会社や複数端末で継続利用する業務アプリとして公開する前に、まず業務データのSupabase移行と、Googleログインのユーザーを組織・ロールへ接続する必要がある。画面の見栄えや細かな操作改善は重要だが、この土台より先に進めると、データ消失・誤った権限・会社間の混在を招く。

公開可否の目安は次のとおり。

| 段階 | 想定利用 | 判定 |
| --- | --- | --- |
| 現在 | 所有者1人がサンプルデータで機能確認 | 可 |
| P0完了後 | 1社での小規模な実データ運用 | 条件付きで可 |
| P1完了後 | 複数担当者・複数会社の運用 | 可 |
| P2完了後 | 継続課金を伴う商用提供 | 目標 |

## 2. 調査範囲と確認結果

### 実施した確認

- React実装、localStorage保存形式、画面上の権限分岐、印刷処理、CSSのレスポンシブ定義を確認した。
- Supabaseの本番テーブル、RLSポリシー、認証ログ、Security/Performance Advisorを確認した。
- GitHub Pagesの直近デプロイ成功と、公開HTMLが最新ビルド資産を返すことを確認した。
- TypeScript型検査、lint、production buildの成功状態を確認した。

共有ブラウザが利用できなかったため、ログイン後画面の見た目・キーボード操作・実機モバイル表示はソースコードからの確認に留まる。本文では、実装で確認済みの事実と、実機で確認すべき項目を分けている。

### 本番DBの実態

| 対象 | 現在の件数 | 読み取り |
| --- | ---: | --- |
| profiles / platform_admins / companies / organizations / memberships | 各1件 | 所有者・会社・組織の初期レコードのみ |
| customers / item_masters / quotes / invoices / activity_logs | 0件 | 業務データは本番DBでまだ利用されていない |

会社のプランだけはcompanies.planを正としており、platform_adminsに登録された所有者だけが変更できる。ここはDBのRLSと列権限で保護されている。

2026-07-21に本番Supabaseへ `add_customer_contact_and_invoice_registration_fields` を適用した。`companies.invoice_registration_no` と、`customers.address`、`phone`、`contact_title`、`invoice_registration_no` はDBに追加済みで、両テーブルのRLSも有効なままである。ただし、現行UIの顧客・自社設定は引き続きlocalStorageを正としており、UIからDBへの同期はR2・R4で接続する。

### すでにできていること

- Google OAuthで本番URLへ戻り、ログインセッションを取得できる。
- 契約プランは一般ユーザーが変更できず、所有者だけが変更できる。
- Supabaseには組織・所属・見積・請求書までを格納できるスキーマとRLS方針がある。
- mainへのpushでGitHub Pagesへ自動デプロイされ、buildもCIで実行される。

## 3. 現在の利用者ウォークスルー

### 3.1 招待された担当者が初めて使う場合

1. ログイン画面でGoogleログインを選ぶ。
2. Google認証が成功すると、業務画面へ入る。
3. 画面にはサンプル会社、サンプル組織、サンプル顧客、サンプル見積が表示される。
4. 担当者は見積を作成・編集・請求書化できるが、保存先はそのブラウザだけである。

問題は、ログイン画面が「招待されたGoogleアカウント」と案内している一方で、アプリ内に招待・参加申請・承認の画面がないことである。認証済みユーザーと業務上の所属は結び付いておらず、初回利用者がどの会社のデータを扱えるのかをアプリが判定していない。

### 3.2 担当者が見積を作成する場合

1. 見積作成で顧客、案件、明細、税区分、数量を入力する。
2. 提出すると一覧へ追加され、見積番号をローカル採番する。
3. 別のブラウザ、別PC、シークレットウィンドウでは同じ見積を開けない。
4. ブラウザ保存領域を消去すると、作成した見積も消える。

ユーザーから見ると「ログインしているのにデータは共有されない」状態である。これは最も誤解を招きやすく、業務利用に進む前に解消すべきである。

### 3.3 管理者が設定を開く場合

1. 設定画面でデモ会社・組織とデモユーザー・ロールを切り替えられる。
2. 見積番号・税率はブラウザ内の設定を編集する。
3. 契約プランだけは所有者がSupabase経由で変更できる。

ここでは、本来の認証ユーザーと画面のデモユーザーが混在している。特に、ログイン直後から「管理者ユーザー」が初期選択されるため、一般ユーザーが閲覧する画面として不正確である。

## 4. 最優先課題（P0）

P0は、実データを入れる前に解決する課題である。UI上で操作できても、データ保全・権限・課金上限が成立しないものを含む。

| ID | 視点 | 問題 | 根拠と影響 | 最初の修正 |
| --- | --- | --- | --- | --- |
| P0-01 | 利用者 / 事業 | 業務データがlocalStorageだけに保存される | 本番DBのcustomers、quotes、invoicesは0件。端末変更・ブラウザ初期化でデータを失い、共同作業できない。 | 顧客、品目、見積、明細、メモ、請求書、活動履歴をSupabaseへ接続する。 |
| P0-02 | セキュリティ / 管理 | 認証ユーザーと画面の会社・組織・ロールが結び付かない | 画面は固定のORGSとUSERSを使用し、ログイン直後もデモ管理者が選択される。誰が何を操作したか保証できない。 | session.user.idからprofiles、memberships、organizationsを取得し、デモ選択UIを撤去する。 |
| P0-03 | 利用者 / 管理 | 招待・所属確認がない | Google認証済みなら業務画面に入れるが、所属組織がない場合の案内・参加申請・承認画面がない。 | 未所属ユーザーはオンボーディング画面へ分岐し、招待トークンまたは参加申請を実装する。 |
| P0-04 | 収益 / セキュリティ | Free上限がフロントエンド判定だけ | プラン自体はDB保護済みだが、見積がlocalStorageであり、件数上限もブラウザ側でしか判定していない。開発者ツールで回避できる。 | DBへの見積作成をRPCまたはトランザクションに寄せ、会社プランと件数をDB側で判定する。 |
| P0-05 | データ保全 | バックアップ、復元、保存失敗通知がない | localStorageの容量超過や解析失敗時にサンプルへ戻る。利用者が気付かないままデータを失う可能性がある。 | DB移行を優先し、移行完了までJSONエクスポート・インポートと保存失敗通知を追加する。 |
| P0-06 | 業務整合性 | 採番と請求書化に同時実行の保証がない | 見積番号・請求書番号がローカル件数と状態に依存する。複数担当者では重複や欠番が起きる。 | 組織単位のDB採番をトランザクション化し、見積作成と請求書化をサーバー側で完結させる。 |

### P0完了の受入基準

- 同じ組織の2ユーザーが別端末で同じ顧客・見積・請求書を見られる。
- 別組織のユーザーはURLやAPIを直接操作しても他社データを読めない。
- 未所属のGoogleユーザーはサンプル画面へ入れず、参加方法が表示される。
- Free会社が上限を超えて見積作成した場合、ブラウザ改変後もDB側で拒否される。
- 見積番号と請求書番号は組織内で重複しない。

### 6.3 セキュリティと運用

- 現在のRLSスキーマは良い出発点だが、画面の業務データがData APIを使っていないため、実運用の保護として働いていない。
- Supabase Security Advisorは漏えいパスワード保護が無効という警告を出している。Googleログインだけなら直ちに影響しないが、メール・パスワード認証を有効にする前に設定する。
- Performance Advisorの未使用インデックスは、業務データが0件のため現時点で削除判断をしない。実データと利用クエリが定まった後に再評価する。
- organization_join_requestsには複数の許可ポリシーに関する性能警告がある。参加申請UIを実装するタイミングで、同じ操作種別のポリシーを統合する。
- Google OAuthを外部利用者へ公開する前に、Google側の公開状態、テストユーザー、ブランド情報、サポート窓口、プライバシーポリシーを確認する。

## 7. UI/UXの課題

### 7.1 情報設計

| 画面 | 確認できた問題 | 改善方針 | 優先度 |
| --- | --- | --- | --- |
| ヘッダー | 実セッションのメールと、デモの会社・ユーザー・ロールが同居する。 | 実ユーザー情報だけを表示し、組織切替は所属組織に限定する。 | P0 |
| ダッシュボード | 見積一覧が9列・多数操作で密度が高く、次にすべきことが弱い。 | KPI、要対応、最近の下書き、期限切れを主にし、詳細操作は行メニューへ集約する。 | P1 |
| 見積フォーム | 明細が横長テーブルで、スマホでは操作負荷が高い。 | PCは表、モバイルは明細カードへ切替え、数量・単価・税区分を段階入力にする。 | P2 |
| 設定 | デモ切替、ローカル設定、DBプランが1画面に混在する。 | 会社設定、組織設定、個人設定、運営管理を分離する。 | P0 |
| 顧客・品目 | 入力欄がplaceholder中心で、必須・入力例・検証結果が分かりにくい。 | 常時ラベル、必須表示、エラー文、保存・取消を明確にする。 | P1 |
| モーダル | プレビュー・メモは開けるが、Escape、フォーカス移動、背面操作防止が実装から確認できない。 | dialog相当のアクセシブルなモーダル、フォーカストラップ、閉じる確認を導入する。 | P2 |

### 7.2 一貫性と安心感

- 「提出する」は社外提出なのか、アプリ内保存なのかが分からない。実際はローカル保存なので、下書き保存、社内確定、PDF出力、メール送付を別の状態として設計する。
- ステータスは一覧で直接変更できるが、変更理由・変更日・担当者の確認が弱い。誤操作対策と履歴表示が必要である。
- 会社名、発行者、プラン、残件数、保存状況を文脈ごとに見せる。現在はユーザーが「どの会社の何を編集しているか」を理解しにくい。
- 画面内メッセージは一時表示で履歴に残らない。通信失敗・競合・権限不足には、原因と再試行導線を持つ通知設計が必要である。

### 7.3 アクセシビリティと実機確認

次は実装から確認できる懸念であり、PC・iPhone・Android・キーボードだけの操作で実機テストを行う。

- フォームの一部が視覚的なplaceholderだけに依存している。
- モーダルにARIA属性、初期フォーカス、フォーカス復帰がない。
- 色だけでステータスや注意を伝える箇所がある。
- 横幅の大きい見積明細・一覧表は、画面幅が狭い端末で読みづらくなる可能性がある。
- ナビゲーションとモーダルのEscape操作、Tab順、スクリーンリーダー読み上げを確認する。

## 8. 推奨する修正順序

### フェーズ0: 実データを入れない安全化

1. [完了] 現行画面を「デモ」または「ベータ」と明示し、契約会社へ正式利用開始を案内しない。
2. [完了] localStorageにある重要データをJSONで退避できるようにする。
3. [一部完了] P0の完了条件、対象会社、移行日、ロール定義を決める。

実装状況: 2026-07-11に、ログイン画面と業務画面へベータ版・ローカル保存の注意表示を追加した。設定画面からlocalStorage主要6キーのJSON書き出し・読み込みができ、保存失敗時は画面メッセージで通知する。

### 9.3 目標アプリ構成

現在の `App.tsx` に型、サンプルデータ、保存処理、全画面を集約した構造は、R1着手時に責務で分割する。目標構成は次を基準とし、機能ごとに画面・型・データアクセス・テストを近くに置く。

```text
src/
  app/              ルーティング、認証初期化、ワークスペース状態
  features/
    onboarding/     未所属、参加申請、招待受理
    customers/      顧客一覧、フォーム、repository
    items/          品目一覧、フォーム、repository
```

構造変更は一度に全画面を書き換えず、R1で土台を作り、R2以降は移行対象の機能だけを `features` 配下へ移す。未移行機能と新機能が同じデータを更新する期間は作らない。

### 9.4 データとAPIの設計ルール

- 業務データの正はSupabaseとし、localStorageは移行用バックアップと個人UI設定以外に使わない。
- Reactコンポーネントから直接テーブルを横断せず、機能ごとのrepositoryまたはserviceを経由する。
- 複数テーブル更新、採番、上限判定、請求書化はRPCで1トランザクションにまとめる。
- DBのsnake_caseと画面のcamelCaseは変換関数で境界を明示し、型アサーションで隠さない。
- スキーマ変更は `supabase/migrations` の履歴として管理し、テーブル、GRANT、RLS、ポリシー、テストを同じ変更単位に含める。
- DB型を生成してフロントエンドの型と照合し、必須列、nullable、enumの不一致をビルド前に検出する。
- 一覧取得には組織IDだけでなくRLSを必須とし、URLやリクエストを書き換えても他組織へ越境できないようにする。
- 通信中、空データ、再試行可能な失敗、権限不足、競合を別状態として画面へ表示する。

### 9.5 新規開発チケットの完了定義

新機能は、次の項目を満たした時だけ完了とする。画面が表示された、または正常系が一度動いた段階では完了にしない。

1. 利用者の操作開始点、成功結果、失敗時の戻り先が画面で確認できる。
2. 必要な型、DB migration、Data API権限、RLS、RPCまたはrepositoryが同じ変更に含まれる。
3. 読み込み中、空状態、入力エラー、通信失敗、権限不足を実装する。
4. 同一組織の成功テストと、別組織・未所属・権限不足の失敗テストを追加する。
5. 金額、税、採番、上限、ステータス遷移などの業務規則を単体テストまたはDBテストで固定する。
6. 既存localStorageデータへの影響、移行方法、ロールバック方法を記録する。
7. `npx tsc -b`、`npm run lint`、`npm run build`を通し、主要画面を実機またはE2Eで確認する。

## 11. 実装時の判断ルール

- 業務データの移行中は、localStorageとDBを二重の正としない。読み取り移行期間を短くし、切替日を決める。
- 権限は画面非表示だけで済ませず、DBのRLS、列権限、必要ならRPCでも強制する。
- プラン・請求・採番・権限変更は監査対象として、変更者と前後値を残す。
- 新しいテーブルは、Data APIの公開設定、GRANT、RLS、ポリシー、RLSテストを同じ変更単位で追加する。
- 会社ごとにDBを分けるのではなく、原則は1つのDBでcompany_idとorganization_idによるテナント分離を行う。法令・契約・大規模顧客の要件が出た時だけ、専用プロジェクトを検討する。
- UI改善は実データの画面を前提に行い、サンプルデータだけで判断しない。主要利用者に見積作成からPDF出力までを観察する。
