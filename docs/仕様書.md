## 1. 文書の目的

本書は、個人事業主・小規模法人向け見積管理アプリ「shonan-dx-app」の現行実装を、運用・保守・今後の開発判断に利用できる形で整理した仕様書である。

本書では、次の2つを区別して記載する。

- **画面で利用可能な現行機能**: React とブラウザの `localStorage` で動作する機能
- **バックエンド準備済み機能**: Supabase のテーブル、Google認証、RLSなど、スキーマまたは接続コードだけが用意されている機能

## 2. システム概要

### 2.1 目的

顧客・品目の登録から見積作成、進捗管理、顧客とのやり取り記録、請求書化、印刷までを1つの画面アプリで管理する。

### 2.2 対象利用者

- 小規模組織の管理者
- 見積作成や顧客対応を行う担当者
- 複数組織を切り替えてデータを確認する利用者

### 2.3 現行の実装状態

| 領域 | 状態 | 備考 |
| --- | --- | --- |
| 見積・顧客・品目・請求書・更新履歴 | 利用可能 | `localStorage` に保存 |
| 見積書・請求書プレビュー、印刷 | 利用可能 | ブラウザ印刷機能を使用 |
| Googleログイン・ログアウト | 接続可能 | Supabase環境変数設定時のみ表示 |
| Supabase業務データ同期 | 未接続 | 現行UIからDBのSELECT/INSERT/UPDATE/DELETEは行わない |
| DBスキーマ、RLS | SQL準備済み | Supabase SQL Editorでの適用が必要 |
| 組織参加申請 | DB仕様のみ | 現行UIは未実装 |

### 3.2 実行モード

#### ローカルモード

`VITE_SUPABASE_URL` または `VITE_SUPABASE_ANON_KEY` が不足している場合に動作する。

- Googleログインボタンを表示しない。
- ヘッダーに「ローカルモード」と表示する。
- 全業務データを `localStorage` から読み書きする。

#### Supabase設定モード

両方のSupabase環境変数が設定されている場合に動作する。

- 未ログイン時は専用ログイン画面だけを表示する。
- ログイン画面ではGoogleログインだけを提供する。
- 初期表示時にSupabaseセッションを取得する。
- 認証状態変更を購読し、表示中のメールアドレスを更新する。
- Google OAuth開始前に、SupabaseでGoogle providerが有効か確認する。
- Google providerが無効な場合は画面内に日本語エラーを表示し、SupabaseのJSONエラー画面へ遷移しない。
- OAuth後のリダイレクト先は、現在のオリジンとパスを連結したURLとする。
- 認証済みの場合だけ業務画面を表示する。
- 業務データは引き続き `localStorage` を使用する。

## 4. 画面構成

### 4.1 ログイン・共通レイアウト

- 未ログイン時
  - アプリ概要
  - Googleログインボタン
  - 認証処理中表示
  - 認証設定・ログインエラー表示
- 認証済みの共通ヘッダー
  - アプリ名
  - 選択中の会社・組織
  - 選択中のデモユーザー・ロール
  - 認証メールアドレス
  - ログアウト操作

- サイドメニュー
  - Dashboard
  - 見積作成
  - 請求書
  - 顧客マスタ
  - 品目マスタ
  - 設定
- メッセージ領域
  - 登録、更新、削除、入力エラーなどの結果を表示
- モーダル
  - 顧客やり取りメモ
  - 見積書・請求書プレビュー

画面幅が1,100px以下の場合はサイドメニューを上部のグリッドへ変更し、720px以下では1列表示へ変更する。表は必要に応じて横スクロールを許可する。

### 4.2 Dashboard

#### KPI

選択中組織について次の値を表示する。

| KPI | 算出方法 |
| --- | --- |
| 見積総額 | 全見積の税込合計額の総和 |
| 成約金額 | ステータスが「成約」の見積額の総和 |
| 請求書 | 作成済み請求書の件数 |
| 要フォロー | 「返答待ち」かつ提出日から10日以上経過した見積件数 |

#### 最新の更新

- 選択中組織の更新履歴を新しい順に最大10件表示する。
- 対象イベントは、見積作成、見積更新、ステータス変更、請求書化、メモ追加とする。
- 保存する更新履歴は全組織を通じて最大200件とする。
- 初回表示で履歴が保存されていない場合は、初期見積・請求書データから最大50件を生成する。

#### 提出済み見積一覧

- 検索対象: 見積番号、取引先名、案件名
- ステータス絞り込み: すべて、返答待ち、成約、請求済
- 並び順: 提出日、更新日、金額、見積番号
- すべて降順で表示する。
- 返答待ちが10日以上の行を警告表示する。
- 提出日と更新日は年月日・時分秒で表示する。

各見積に対して次の操作を提供する。

| 操作 | 動作 |
| --- | --- |
| ステータス変更 | 返答待ち、成約、請求済を直接選択し、更新日時と履歴を更新 |
| プレビュー | 見積書プレビューを表示 |
| 編集 | 内容を見積入力画面へ展開し、同じ見積番号で更新 |
| 複製 | 明細を複製し、新規見積として入力画面へ展開 |
| 請求書化 | 対応する請求書を作成し、見積を請求済へ変更 |
| メモ | 顧客やり取りメモの追加・履歴表示 |
| 削除 | 確認後、見積と紐づくローカル請求書を削除 |

### 4.3 見積作成・編集

#### 入力項目

| 項目 | 仕様 |
| --- | --- |
| 顧客 | 選択中組織の顧客マスタから選択 |
| 案件名 | 任意。空欄で提出した場合は「無題の案件」 |
| 明細 | マスタ明細または自由明細を1件以上保持 |
| 品目名 | マスタ明細は選択値、自由明細は直接入力 |
| 税区分 | 課税または非課税 |
| 単価 | 0以上の数値 |
| 数量 | 1以上の数値。現行UIでは整数・小数の入力制限を明示していない |
| メモ | 見積書の備考欄へ表示 |

#### 金額計算

1. 明細金額 = 単価 × 数量
2. 小計 = 全明細金額の合計
3. 課税対象 = 課税明細金額の合計
4. 消費税 = `Math.round(課税対象 × 税率 ÷ 100)`
5. 合計 = 小計 + 消費税

非課税明細は小計には含め、消費税計算の対象外とする。通貨表示は日本円、桁区切りあり、小数なしとする。

#### 見積番号

- 形式: `{prefix}-{year}-{3桁以上の連番}`
- 初期値: `Q-2026-042`
- 新規提出時に次番号を1増加する。
- 編集時は既存番号を維持する。
- 年は現行画面から変更できず、保存済み設定値を使用する。

#### 登録制限

- Freeプランは、選択中組織の見積件数が環境変数 `VITE_APP_FREE_QUOTE_LIMIT` 未満の場合のみ新規作成できる。
- 環境変数がない場合の上限は20件とする。
- Proプランは件数制限を適用しない。
- 既存見積の編集には上限を適用しない。

### 4.4 請求書

- 見積からのみ作成できる。
- 作成時に見積額、顧客名、見積IDを保持する。
- 請求書番号は `INV-{作成時の西暦}-{全請求書件数+1の3桁連番}` とする。
- 同じ見積に既存請求書がある場合、新規作成せず既存請求書のプレビューを表示する。
- 請求書化した見積は「請求済」へ変更し、請求書番号を保持する。
- 請求書一覧では、番号、取引先、金額、作成日時を表示する。

### 4.5 顧客マスタ

| 項目 | 必須 | 備考 |
| --- | --- | --- |
| 顧客名 | 必須 | 前後空白を除去して保存 |
| 担当者 | 任意 | 文字列 |
| メール | 任意 | 形式検証なし |
| メモ | 任意 | 文字列 |

- 追加は管理者・担当者のどちらでも画面上実行できる。
- 編集・削除ボタンは管理者にのみ表示する。
- 見積で使用中の顧客は削除できない。
- 顧客名の重複は許可する。

### 4.6 品目マスタ

| 項目 | 必須 | 備考 |
| --- | --- | --- |
| 品目名 | 必須 | 前後空白を除去して保存 |
| カテゴリ | 任意 | 文字列 |
| 単価 | 必須 | 0以上 |
| 単位 | 任意 | 初期値「式」 |
| 税区分 | 必須 | 課税または非課税 |

- 追加は管理者・担当者のどちらでも画面上実行できる。
- 編集・削除ボタンは管理者にのみ表示する。
- 入力中の見積または保存済み見積で使用中の品目は削除できない。
- 品目名の重複は許可する。

### 4.7 設定

- デモ会社・組織を切り替える。
- デモユーザー・ロールを切り替える。
- 見積番号プレフィックス、次番号、税率を変更する。
- 契約プランはSupabaseの会社情報から取得し、一般ユーザーには参照専用で表示する。
- 運営管理者だけが契約会社のFree/Proプランを変更できる。
- 組織切り替え時は、その組織で最初に見つかった顧客と品目を入力フォームへ設定し、編集中状態を解除する。

現行の見積番号・税率設定は組織別ではなく、ブラウザ内で1つの設定オブジェクトとして保存する。契約プランとFree上限は `companies` を正とし、Supabaseから取得する。Supabaseスキーマ上の `quote_number_settings` は組織別だが、UIとは未接続である。

### 4.8 メモ

- 見積ごとに複数件保存する。
- 空白だけのメモは追加しない。
- メモ本文、選択中のデモユーザー名、追加日時を保持する。
- 新しいメモを先頭に表示する。
- 追加時に見積の更新日時を更新し、更新履歴を追加する。

### 4.9 プレビュー・印刷

- 入力中または保存済み見積から見積書をプレビューできる。
- 請求書一覧または請求書化操作から請求書をプレビューできる。
- 書類には番号、発行日、宛先、案件名、発行者、明細、税区分、単価、数量、金額、備考、集計を表示する。
- 「PDF化 / 印刷」は `window.print()` を呼び出す。
- 印刷時は操作UIを非表示にし、A4縦向けの書類部分だけを出力する。
- PDFファイルをアプリ内で直接生成・保存する機能は持たない。

## 5. 権限仕様

### 5.1 現行UI権限

| 操作 | 管理者 | 担当者 |
| --- | --- | --- |
| ダッシュボード閲覧 | 可 | 可 |
| 見積作成・編集・複製・削除 | 可 | 可 |
| ステータス変更・請求書化・メモ追加 | 可 | 可 |
| 顧客・品目の追加 | 可 | 可 |
| 顧客・品目の編集・削除 | 可 | 不可（ボタン非表示） |
| 組織・ユーザー切り替え | 可 | 可 |
| 見積番号・税率変更 | 可 | 可 |
| 契約プラン変更 | 不可 | 不可 |

契約プラン以外のUI権限は画面内のデモユーザー選択値に基づく表示制御であり、Supabaseの認証ユーザーやDB権限とは連動していない。契約プラン変更は `platform_admins` に登録された運営管理者だけに表示し、DBのRLSと列権限でも制限する。

### 5.2 Supabase DB権限

- 公開スキーマ内の全業務テーブルでRLSを有効にする。
- `authenticated` と `service_role` にテーブル操作権限を明示的に付与する。
- 利用者は所属組織の業務データだけを参照・更新できる。
- プロフィールは本人だけが参照・登録・更新できる。
- 所属メンバーは組織・会社情報を参照できる。
- 所属メンバーは会社の契約プランを参照できるが変更できない。
- `platform_admins` に登録された本人は全会社を参照し、`companies.plan` だけを変更できる。
- 一般ユーザーは運営管理者を追加・変更・削除できない。
- 所属メンバーは顧客、品目、見積設定、見積、見積明細、見積メモ、請求書を管理できる。
- 更新履歴は所属メンバーが参照・追加でき、更新・削除は許可しない。
- 組織ごとに管理者は1人だけ登録できる。
- ブラウザ権限では管理者メンバー行を追加・変更・削除できない。

## 6. データ保存仕様

### 6.1 localStorage

名前空間は `estimate-management-v3` とし、次のキーを使用する。

| キー | 保存内容 |
| --- | --- |
| `estimate-management-v3:customers` | 顧客配列 |
| `estimate-management-v3:items` | 品目配列 |
| `estimate-management-v3:quotes` | 見積、明細、メモの配列 |
| `estimate-management-v3:invoices` | 請求書配列 |
| `estimate-management-v3:activities` | 更新履歴配列 |
| `estimate-management-v3:settings` | 税率、番号設定 |

- JSONとして保存する。
- 過去データの `settings.plan` は読み込み時に無視し、以後の保存対象から除外する。
- 読み込みまたはJSON解析に失敗した場合は初期サンプルデータを使用する。
- 過去の品目に `taxKind` がない場合は `taxable` を補完する。
- 初期の非課税品目「立替交通費」が保存済み品目に存在しない場合は補完する。
- localStorageの容量超過など、保存時エラーを画面通知する処理はない。
- ブラウザ、ドメイン、端末をまたぐ同期は行わない。

### 6.2 フロントエンド主要データ型

| データ | 主な属性 |
| --- | --- |
| 組織 | ID、会社名、組織名 |
| ユーザー | ID、表示名、メール、ロール、組織ID |
| 顧客 | ID、組織ID、名称、担当者、メール、メモ |
| 品目 | ID、組織ID、名称、カテゴリ、単価、単位、税区分 |
| 見積明細 | ID、品目ID、名称、単価、数量、単位、税区分、自由入力フラグ |
| 見積 | ID、組織ID、番号、顧客、案件、金額、状態、作成・更新日時、備考、明細、メモ、請求書番号 |
| 請求書 | ID、組織ID、番号、見積ID、顧客名、金額、作成日時 |
| 更新履歴 | ID、組織ID、種別、タイトル、説明、作成日時 |
| 設定 | 税率、プレフィックス、年、次番号 |

## 7. Supabaseデータモデル

### 7.1 階層と関連

```text
companies
└─ organizations
   ├─ organization_memberships ─ profiles ─ auth.users
   ├─ organization_join_requests
   ├─ customers
   ├─ item_masters
   ├─ quote_number_settings
   ├─ quotes
   │  ├─ quote_items
   │  ├─ quote_interaction_notes
   │  └─ invoices
   └─ activity_logs
```

### 7.2 テーブル一覧

| テーブル | 用途 | 主な制約 |
| --- | --- | --- |
| `companies` | 会社と契約プラン | Free上限を保持 |
| `organizations` | 会社配下の組織 | 会社削除時に連鎖削除 |
| `profiles` | Authユーザーの業務プロフィール | `auth.users.id` と同一ID |
| `platform_admins` | 契約プランを管理する運営管理者 | 本人だけ参照可、ブラウザから追加・変更・削除不可 |
| `organization_memberships` | 組織所属とロール | 組織・ユーザー重複不可、組織内adminは1人 |
| `organization_join_requests` | 組織参加申請 | 同一組織・ユーザーのpendingは1件 |
| `customers` | 顧客マスタ | 組織に所属 |
| `item_masters` | 品目マスタ | 単価0以上 |
| `quote_number_settings` | 組織別番号・税率 | 組織ごとに1件、次番号1以上 |
| `quotes` | 見積ヘッダー | 組織内見積番号は一意、金額0以上 |
| `quote_items` | 見積明細 | 数量0超、単価0以上 |
| `quote_interaction_notes` | 見積のやり取りメモ | 見積削除時に連鎖削除 |
| `invoices` | 請求書 | 組織内請求書番号は一意 |
| `activity_logs` | 更新履歴 | 組織・作成日時で索引 |

### 7.3 列挙値

| 型 | 値 |
| --- | --- |
| `user_role` | `admin`, `member` |
| `quote_status` | `pending`, `won`, `invoiced` |
| `plan_type` | `free`, `pro` |
| `tax_kind` | `taxable`, `exempt` |
| `activity_kind` | `quote-created`, `quote-updated`, `status-updated`, `invoice-created`, `memo-added` |
| `join_request_status` | `pending`, `approved`, `rejected`, `canceled` |

## 8. 認証・初期セットアップ・組織参加

### 8.1 Google認証

1. SupabaseでGoogle providerを有効化する。
2. Site URLとRedirect URLにローカル、GitHub Pages、本番の各URLを登録する。
3. 未ログイン利用者へ専用ログイン画面を表示する。
4. Google providerの有効状態を確認してからGoogle OAuthを開始する。
5. リダイレクト後、Supabaseセッションを取得する。
6. セッション取得後に業務画面を表示する。

フロントエンドに設定するキーはpublishable keyまたは互換用anon keyとし、`service_role` keyは設定しない。

### 8.2 初回管理者作成

1. Googleログインにより `auth.users` を作成する。
2. `supabase/initial_setup.sql` のメール、表示名、会社名、組織名を実値へ変更する。
3. Supabase SQL Editorで実行する。
4. プロフィール、会社、組織、管理者所属、見積番号設定を作成する。

初期設定SQLは、同名会社・同一会社内の同名組織がある場合に既存行を再利用する。対象プロフィールがすでにある場合は表示名とメールを更新する。

### 8.3 組織参加フロー

1. 追加ユーザーがGoogleログインする。
2. 本人のプロフィールを作成する。
3. 未所属組織に対して `pending` の参加申請を作成する。
4. 管理者が申請を `approved` または `rejected` に更新する。
5. 承認時、管理者が対象ユーザーを `member` として所属へ追加する。
6. 申請者は未処理の申請を `canceled` に変更できる。

このフローを操作する画面は現行フロントエンドに存在しない。

## 11. 非機能仕様

### 11.1 対応環境

- ES2023とDOM APIを利用できるモダンブラウザを対象とする。
- JavaScript、`localStorage`、CSS Grid、ブラウザ印刷機能が有効であることを前提とする。
- UI言語は日本語、通貨は日本円、日付表示ロケールは `ja-JP` とする。

### 11.2 セキュリティ

- フロントエンドへ秘密鍵やSupabase `service_role` keyを配置しない。
- DB適用時は全公開テーブルでRLSを有効にする。
- 組織所属判定と管理者判定は `app_private` スキーマの関数を介して行う。
- 現行ローカルデータはブラウザ利用者が開発者ツールから参照・変更できるため、機密情報の正式保管先とはしない。
- Supabase設定モードでは、Googleログイン済みの場合だけ業務画面を表示する。

### 11.3 データ整合性

- フロント側のIDは時刻と乱数から生成する文字列であり、DBのUUIDとは形式が異なる。
- 見積削除時はローカル請求書も削除するが、更新履歴は削除しない。
- 顧客名や品目内容は見積へスナップショットとして保持するため、マスタ更新後も既存見積表示は自動更新しない。
- 請求書金額は作成時の見積額を保持し、その後の見積編集による自動再計算は行わない。

## 12. 既知の制約と未実装事項

1. 業務データはSupabaseへ保存されず、端末間・利用者間で共有されない。
2. 認証ユーザーと画面上のデモユーザー、組織、ロールは連動していない。
3. 組織参加申請、承認、メンバー管理のUIはない。
4. DBの会社別プラン・Free上限は画面と連動するが、見積件数の上限判定は現時点ではフロントエンド上のみで、DBへの見積保存時の上限制御は未実装である。
5. 画面の設定値は組織別ではなく全組織共通である。
6. 請求書番号は全組織のローカル請求書件数から採番するため、削除後の再採番や組織別一意性を保証しない。
7. 見積番号の同時採番制御はなく、複数端末利用時の競合を考慮していない。
8. メール形式、文字数、税率上限、金額上限などの詳細な入力検証はない。
9. localStorageの明示的なエクスポート、インポート、バックアップ、初期化機能はない。
10. 自動テストは設定されていない。
11. PDFはブラウザ印刷経由であり、サーバー保存、送信、電子帳簿保存対応はない。

## 13. Supabase連携を本実装する際の受入条件

現行DB設計を実運用へ接続する場合、少なくとも次を満たす必要がある。

- ログイン必須のルートまたは画面ガードを実装する。
- 認証ユーザーのプロフィール・所属組織・ロールをDBから取得する。
- 顧客、品目、見積、明細、メモ、請求書、履歴、設定のCRUDをSupabaseへ切り替える。
- 見積・請求書番号の採番をDBトランザクションで競合なく行う。
- localStorage既存データの移行、重複判定、ロールバック方法を定義する。
- フロントエンド型とDB列挙値の大文字・小文字差を変換する。
- 組織参加申請と管理者承認画面を実装する。
- RLS下で管理者・担当者・未所属者の正常系と拒否系を検証する。
- 顧客削除、見積削除、請求書との関連についてDB外部キーとUI動作を一致させる。
- 型検査、lint、ビルド、自動テストをCIで実行する。

## 14. 変更管理方針

- 本書は現行実装を正とするAs-Is仕様書である。
- UI、TypeScript型、localStorage形式、Supabaseスキーマ、RLS、採番規則を変更した場合は、本書の該当章も同時に更新する。
- localStorage形式を変更する場合は、既存の `estimate-management-v3:*` データを読み込める補完またはマイグレーションを用意する。
- DBスキーマ変更は、既存データへの影響、移行、後方互換、ロールバック方法を確認してから適用する。
