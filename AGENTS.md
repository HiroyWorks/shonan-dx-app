# Codex 作業ルール

- このリポジトリのコミットメッセージは必ず日本語で書く。
- 1行目は変更内容がすぐ分かる短い要約にする。
- 英語の定型句だけのコミットメッセージは使わない。

## 変更前の確認

- 変更対象ファイルと、そのファイルが参照している関連モジュールを先に確認する。
- 既存の型定義、データ構造、props、state、localStorage の保存形式を把握してから編集する。
- JSON、設定ファイル、Supabase schema、TypeScript の型など、構造が決まっているファイルは編集前後で構文が壊れていないか確認する。

## 文字コード

- 日本語を含むファイルは、編集前に文字コードと現在の表示状態を確認する。
- PowerShell やターミナル表示で文字化けして見えても、ファイル本体が UTF-8 で正しく保存されている場合があるため、表示だけで判断しない。
- 日本語テキストを含むファイルを書き換えるときは UTF-8 を維持し、Shift_JIS や別エンコーディングへ変換しない。
- 文字列全体を不用意に再生成せず、必要な箇所だけを最小差分で編集する。
- 編集後はアプリ上の日本語表示が壊れていないか、少なくともビルドまたは該当箇所の確認を行う。

## 大幅変更時の進め方

- 大幅な仕様変更や構造変更が必要な場合は、既存コードへ無理に継ぎ足さない。
- 現状の構造に合わせ続けることで保守性や安全性が下がる場合は、影響範囲を確認したうえで再設計・分割実装を選ぶ。
- UI変更、型変更、保存データ変更、DB/API変更はできるだけ責務を分け、必要に応じて段階的にコミットする。
- 既存データを壊す可能性がある変更では、移行方法、後方互換、ロールバックしやすさを先に考える。
- 判断に迷うほど影響が大きい場合は、実装前に選択肢とリスクを整理して確認する。

## 型と構造の安全

- TypeScript の型を追加・変更するときは、関連する state、関数引数、保存データ、表示コンポーネントまで追跡する。
- `any` や型アサーションで無理に通さず、既存の型に沿って安全に拡張する。
- JSON や設定オブジェクトを編集するときは、キー名、必須項目、配列/オブジェクトの形を壊さない。
- 既存の localStorage データ形式を変える場合は、古いデータが残っていても動くように補完・マイグレーション処理を用意する。
- 変更後は原則として `npx tsc -b`、`npm run lint`、`npm run build` の順で確認する。
